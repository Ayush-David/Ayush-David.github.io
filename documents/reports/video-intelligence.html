A Data-Driven Intelligence Layer for Human-Centric Industry 5.0 Manufacturing

Ayush David Feb – Apr 2025 Bengaluru, India Research Internship
Computer Vision Multi-Engine Inference Kafka Microservices RAG / LLM Orchestration Time-Series Analytics Docker

A microservice intelligence layer for electronics manufacturing plants, combining live multi-camera video analytics with a manufacturing process analytics workbench and a natural-language AI Copilot — deployed and validated on campus infrastructure at IISc ARTPARK.

The Problem

Modern manufacturing lines generate enormous volumes of machine telemetry — SPI inspection coordinates, reflow zone temperatures, pick-and-place placement logs — but most of it is siloed in proprietary controllers or exported to flat CSVs and never correlated across stages. Floor supervisors make quality and downtime decisions on end-of-shift review rather than live evidence, and there's no system that watches camera feeds for safety/quality events and machine telemetry for process drift and surfaces both to an operator in a form they can actually query.

Architecture
  • 01
    Data ingestion
    RTSP camera streams converted to MJPEG/WebRTC; a camera registry service (SQLite, CSV/Excel bulk import) handles metadata and credentials. SPI, Reflow, and P&P production datasets ingested through format-specific parsers and merged into a unified analytical dataset.
  • 02
    Stream processing & inference
    A plug-and-play scene-understanding service runs four concurrent engine types — YOLO object/person detection (ONNX Runtime), InsightFace recognition, a VLM for scene description, and action recognition — each defined by a manifest, addable without touching core orchestration. A scene-change filter with burst buffering forwards only frames with meaningful change.
  • 03
    Storage
    Four stores, one per data shape — below.
  • 04
    AI query & alerting
    An MCP-pattern query service routes operator questions to TimescaleDB retrieval, Qdrant semantic search, or live scene analysis, then synthesizes an answer via a multi-provider LLM backend (OpenAI / Google / local Ollama). The alert service parses plain-language alert definitions into structured conditions + embeddings, then matches them against live Kafka events in real time.
  • 05
    Operator interface
    A FastAPI + Jinja2 dashboard for camera management, event browsing, alerts, and reports, plus a chat-first Factory Copilot that maps free-text operator intent to specific dashboard widgets and deep links.

The platform deployed cleanly under Docker Compose with health-checked, independently restartable services; a watchdog confirmed automatic recovery from induced failures. The harder problem wasn't any single engine — it was latency discipline across the full pipeline (capture → inference → Kafka → DB write → WebSocket push) and keeping the Copilot's Qdrant-retrieved context fresh enough that answers didn't quietly go stale. Both were solved with tuned producer/consumer and burst-buffer parameters, and a faster embedding-refresh cycle with a staleness indicator on Copilot responses.

What I'd Build Next

Move the manufacturing analytics path from batch CSV ingestion to real-time streaming, so process analytics runs at the same latency tier as video. Add multi-turn memory to the Copilot so operators can refine a query across a session. And the real test: deploy at an active production facility, not a campus testbed, to see if the architecture holds under real data diversity and load.
